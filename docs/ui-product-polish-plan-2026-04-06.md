# 客户端产品质感改造方案

最后更新：2026-04-06

这份文档回答四件事：

- 为什么当前产品会持续给人一种“像 demo”的感觉
- 这不是单点视觉问题，而是哪些系统性问题叠加导致的
- 应该按什么顺序改，才能先消除“拼装感”，再提升整体质感
- 每个阶段的产出、边界和验收标准是什么

## 1. 当前判断

当前前端的问题不是“功能太少”，而是“产品气质没有收口”。

仓库里已经有：

- 品牌名称、口号与基础品牌常量
- 首页、工作台、聊天、知识库等完整核心页面
- 自定义字体接入
- 一份更完整的品牌预览页面

但真正运行时的 UI 仍然表现出明显的“脚手架默认感”和“模块各自为战”的状态。结果不是“做不出来”，而是“像若干个还不错的页面拼在一起”，没有形成一个成熟产品应有的统一视觉与信息系统。

## 2. 问题拆解

### 2.1 品牌系统没有真正进入运行时界面

当前仓库同时存在两层视觉定义：

- 运行时主题 token：`packages/client/src/index.css`
- 品牌预览稿：`packages/client/public/brand-theme-preview.html`

问题不在于没有品牌，而在于品牌没有真正接管产品：

- `index.css` 仍以中性灰为核心，接近默认组件库气质
- `brand-theme-preview.html` 里已经定义了更完整的品牌绿、深浅 surface、按钮和暗色模块规则，但这些规则没有回流到真实页面
- 侧边栏、聊天输入等局部组件又额外写死了一套颜色，进一步加剧视觉漂移

结果是首页、工作台、聊天页虽然都能用，但看起来不像同一个产品。

### 2.2 基础组件保留了明显的脚手架气味

当前 `Button`、`Card` 等基础组件仍然以通用 shadcn 风格为主，表现为：

- 圆角、阴影、描边和 hover 反馈过于保守
- 各业务页面大量使用同一套 `border + bg-card + hover:bg-accent` 组合
- 页面差异更多来自内容不同，而不是组件语言本身不同

这种实现效率很高，但也最容易让界面停留在“默认后台模板”的观感。

### 2.3 页面有内容，但没有形成“真实产品感”

首页、控制台、知识库页都存在类似问题：

- 有标题、按钮、卡片和统计
- 有完整功能跳转
- 也有少量视觉装饰

但缺少能够传递“这个产品正在被真实使用”的信息密度。

典型表现：

- 首页更像一张产品说明页，而不是一个成熟产品入口
- Dashboard 更像欢迎页，而不是工作台
- 核心页面缺少最近活动、处理状态、关键指标、真实对象列表等产品运营信息

用户看到的是“功能介绍”，不是“系统状态”。

### 2.4 同类对象存在多套视觉语言

知识库相关视图已经出现了至少两套不同风格：

- 一套偏彩色、强调 hover 与浮起感
- 一套偏克制、以灰白卡片和弱状态为主

这会产生两个直接后果：

- 视觉语言不稳定，用户很难建立熟悉感
- 产品看起来像“迭代中间态”，不像完成过一次统一设计收口

### 2.5 视觉层级存在，但不够强

当前界面的主要层级依赖：

- 字号差异
- 文本颜色差异
- 少量描边和留白

缺少更明确的层次锚点：

- 哪些区域是品牌层
- 哪些区域是工作区
- 哪些区域是可操作对象
- 哪些区域是系统反馈和状态

所以页面虽然整洁，但显得“轻、薄、平、像占位稿”。

## 3. 目标

这次改造的目标不是“做得更花”，而是让产品更像一个完成过设计收口的知识工作台。

本方案的目标：

- 建立统一的运行时品牌 token，并替换现有漂移的颜色系统
- 让首页、控制台、聊天页、知识库页共享同一套产品壳和视觉语言
- 提升工作台与核心业务页的信息密度，减少“欢迎页”与“说明页”感
- 统一同类业务对象的卡片、列表、表格和空态样式
- 让视觉重心从“组件库默认样式”转向“Groundpath 自己的产品气质”

本方案不追求：

- 一次性重做所有页面
- 引入大量装饰性动效
- 为了“高级感”牺牲信息可读性和可维护性
- 在没有视觉基础设施收口前，先做零散局部美化

## 4. 设计方向

建议把产品气质收敛到“可信知识工作台”，而不是“通用 AI 工具”。

核心关键词：

- 可信
- 稳定
- 有结构
- 有信息密度
- 克制但不平庸

这意味着：

- 主色应该承担品牌识别和关键操作，不应到处泛滥
- 深浅主题都要有明确的 surface 分层，而不是简单黑白反转
- 卡片应有层次，但不靠大阴影和夸张浮动制造存在感
- 首页要展示真实产品片段，控制台要展示真实工作流状态

## 5. 总体策略

改造顺序不能从零散页面开始，而要先收口“底座”，再抬高“产品感”。

### 5.1 先改视觉底座

优先建立统一 token 与基础组件语言：

- 颜色
- 字体层级
- radius
- border
- shadow
- focus ring
- 主按钮 / 次按钮 / 幽灵按钮 / 卡片

原因：

- 如果底座不统一，后续所有页面改造都会继续发散
- 页面做得越多，返工成本越高

### 5.2 再改产品壳

统一应用级结构：

- 侧边栏
- 顶部工具区
- 内容画布
- 聊天输入区
- 模块间留白与层次

原因：

- 应用壳是用户感知“这是不是一个完整产品”的第一层
- 只改内容页，不改外壳，整体气质不会真正变化

### 5.3 最后抬高页面信息密度

在统一的底座和应用壳之上，再改：

- Dashboard
- Home
- Knowledge Bases
- 其他设置和资料页面

这样可以保证每个页面不是单独美化，而是在同一套系统下生长出来。

## 6. 分阶段改造计划

### 6.1 第 0 阶段：视觉基线冻结

目标：

- 明确运行时品牌 token 的唯一来源
- 确定产品的基础视觉语言

改造范围：

- `packages/client/src/index.css`
- `packages/client/src/components/ui/button.tsx`
- `packages/client/src/components/ui/card.tsx`
- 必要时补充通用 surface / layout utility

核心动作：

- 以 `brand-theme-preview.html` 为参考，回收品牌绿与深浅 surface 规则
- 删除或收敛当前分散的硬编码颜色
- 重新定义按钮与卡片的默认气质
- 明确 hover、focus、active 的统一规则

完成标准：

- 主色、surface、边框、文字层级在全局统一
- 新页面默认不会再自动长成“灰白模板”
- 基础组件一眼能看出不是原生默认皮肤

### 6.2 第 1 阶段：应用壳重构

目标：

- 让侧边栏和主内容区成为稳定的产品识别锚点

改造范围：

- `packages/client/src/components/layout/AppLayout.tsx`
- `packages/client/src/components/layout/AppSidebar.tsx`
- `packages/client/src/components/chat/ChatInput.tsx`
- 与壳层直接相关的少量 toolbar / shell 组件

核心动作：

- 把侧边栏从“功能列表容器”升级为“产品导航锚点”
- 统一侧边栏、内容画布、滚动容器之间的层次关系
- 取消局部散落的品牌绿硬编码
- 让聊天输入区与整体产品壳保持一致，而不是像独立控件

完成标准：

- 登录后的所有主页面看起来属于同一套应用壳
- 侧边栏、内容区、工具栏有清晰层级
- 聊天页不再像单独拼接进去的模块

### 6.3 第 2 阶段：Dashboard 从欢迎页升级为工作台

目标：

- 让 dashboard 成为“正在运行的产品首页”

改造范围：

- `packages/client/src/pages/DashboardPage.tsx`
- `packages/client/src/components/dashboard/*`

核心动作：

- 保留欢迎语，但降低其占比
- 强化关键指标、最近对象、待处理状态、系统观测入口
- 把已有的 `StructuredRagOverview` 纳入主页面信息流
- 补足“最近活动 / 当前状态 / 下一步动作”三类信息

完成标准：

- dashboard 打开后首先看到系统状态，而不是空泛欢迎语
- 页面能够回答“现在系统里发生了什么”
- 用户能自然进入下一个关键任务

### 6.4 第 3 阶段：首页从说明页升级为产品入口

目标：

- 首页不再只讲概念，而要证明产品已经成型

改造范围：

- `packages/client/src/pages/Home.tsx`
- `packages/client/src/pages/About.tsx`

核心动作：

- 保留品牌文案，但增加真实产品预览
- 展示真实工作流、引用式回答、知识库结构等产品片段
- 让 hero 的视觉重心从“文字介绍”转向“产品截图式体验”
- 减少“3 张通用能力卡 + 3 步流程”的模板化结构占比

完成标准：

- 首页能在数秒内传递产品气质，而不是只传递功能点
- 访客能形成对产品成熟度的第一印象

### 6.5 第 4 阶段：业务对象视图统一

目标：

- 同类对象只保留一套视觉语言

改造范围：

- `packages/client/src/pages/knowledge-bases/*`
- `packages/client/src/components/knowledge-bases/*`
- 文档卡片、空态、表格、列表等相关对象视图

核心动作：

- 合并知识库卡片的多套风格
- 统一空态、无结果态、创建卡片、列表行、表格行语言
- 明确对象卡片的状态表达方式，不再依赖随机式局部装饰

完成标准：

- 同一业务对象在不同页面中保持统一气质
- 页面切换时不再出现“像另一个产品模块”的感觉

## 7. 具体落地原则

### 7.1 颜色只保留一套来源

- 品牌色以统一 token 输出
- 局部组件禁止继续写死 `#10a37f` 一类值
- 深浅模式都从 token 派生，不手工各写一套风格

### 7.2 组件风格不能再靠业务页面自行拼

- 业务页面只组合，不重新发明按钮和卡片语言
- 如果确实需要特殊容器，应抽象为新的通用 surface 组件

### 7.3 信息密度优先于装饰密度

- 优先补真实数据、状态和对象关系
- 谨慎使用大阴影、玻璃拟态、过强渐变和无意义动画
- “高级感”来自结构和秩序，不来自堆效果

### 7.4 页面必须形成明确的阅读路径

每个核心页面都需要回答三件事：

- 当前页面的主任务是什么
- 当前系统或对象状态是什么
- 下一步最合理的操作是什么

如果页面只能回答第一件事，通常就会像 demo。

## 8. 风险与约束

### 8.1 风险

- 只改颜色、不改结构，最终只会从“灰 demo”变成“有品牌色的 demo”
- 只改首页、不改登录后应用壳，用户进入主系统后仍会失望
- 同时大改所有页面，容易造成风格反复摇摆，难以收口

### 8.2 约束

- 当前仓库已经有较多客户端测试，改造时要尽量保持结构稳定
- 这次工作以客户端质感提升为主，不涉及 API 语义变更
- 不应该为了视觉重构破坏已有路由、交互和可访问性基础

## 9. 验收标准

当以下条件同时满足时，可以认为“demo 感”被显著压下来了：

- 首页、Dashboard、聊天页、知识库页看起来明显属于同一个产品
- 品牌色、surface、卡片、按钮和侧边栏语言一致
- Dashboard 具有真实工作台的信息密度，而不是欢迎页结构
- 首页能够展示真实产品体验，不只是功能说明
- 同类业务对象不再存在两套及以上主要视觉语言
- 用户第一次进入产品时，不会产生“这是个搭好的模板壳子”的直觉

## 10. 推荐执行顺序

建议严格按以下顺序推进：

1. `index.css` + `button/card`
2. `AppLayout/AppSidebar` + `ChatInput`
3. `DashboardPage`
4. `Home` / `About`
5. `knowledge-bases` 与相关对象视图统一

顺序不能反。

原因：

- 越靠前的项越像基础设施
- 越靠后的项越依赖前面的视觉基线
- 如果反过来做，返工会明显增多

## 11. 当前阶段结论

当前最合理的下一步不是继续讨论“缺什么风格”，而是直接进入第 0 阶段与第 1 阶段：

- 先完成运行时 token 与基础组件收口
- 紧接着完成应用壳重构

只有这两步完成后，后续页面改造才会开始真正摆脱 demo 感。

## 12. API 变更说明

本方案是客户端视觉与信息架构改造文档，不涉及 API 路由、请求参数、响应结构、权限语义或分页过滤语义变更，因此本阶段不需要更新 `docs/api-changelog/v1.md`。
