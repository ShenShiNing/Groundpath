## 5. 现有自动化用例检查结论

### 5.1 已覆盖较好的区域

1. Auth service 主流程与异常分支。
2. Document service（CRUD/上传/版本/回收站）主要逻辑。
3. Document AI service（摘要/分析/生成）大部分分支。
4. 安全工具（JWT/Cookie/Refresh Token）。
5. Chat service + controller（conversation/message）核心分支与边界条件已覆盖。
6. Knowledge Base/Document/Chat/Logs/RAG/Storage 路由已具备 HTTP 行为级测试，覆盖鉴权、参数、分页、UUID、签名、路径穿越等边界。
7. OAuth/邮箱验证码/用户资料/Document AI 本轮补齐 HTTP 行为级测试，覆盖 CSRF、回调缺参、验证码格式、限流、头像超限、AI 请求参数上下界等边界。
8. 当前后端“已实现核心功能域”均至少有 route 级或 HTTP 行为级自动化覆盖，形成上线前回归闭环起点。

### 5.2 缺口清单（建议优先补测）

1. `ops` 当前为单元级覆盖，仍缺少进程级集成验证（真实 `SIGTERM/SIGINT`、监听端口关闭、资源释放顺序）。
2. `chat` 模块 repository 与 `chat.service` 深层失败分支仍是高风险区（消息持久化、流式中断恢复、外部依赖失败降级）。
3. `llm/vector/storage/rag` 仍缺少外部依赖异常注入与重试/降级策略验证（timeout、5xx、网络抖动）。
4. 前端缺少页面级测试和 E2E：
   - 路由总数约 `20`，当前仅少量 store/lib 单测。
5. 缺少迁移后数据完整性自动化检查。
