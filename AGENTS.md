# Repository Guidelines

## 语言

- 与协作者沟通默认使用中文。

## Git 工作流

- 每个功能必须在独立分支上开发，禁止在同一分支上混合多个功能。
- 每完成一个阶段性变更后，自动创建 Git commit 并推送到远程仓库。
- 使用 Conventional Commits（`feat(scope): …`、`fix(scope): …`），需要时附上清晰的 body。
- 提交信息中禁止包含工具署名（如 "generated by Claude Code/Codex"）。
- 每次发生 API 相关变化时，必须同步更新 `docs/api-changelog/v1.md`；如暂不需要记录，必须在 PR 或提交说明中明确原因。
- API 相关变化至少包括：路由/方法变更、请求参数或请求体变更、响应结构或错误结构变更、分页/过滤/排序语义变更、鉴权/权限/限流语义变更、兼容层新增或移除。

## 架构偏好

- 多步流程（上传/删除/恢复/版本管理）必须在单一 service 中编排，副作用保持配对（删除 → 计数器递减；恢复 → 递增）。
- 计数器/统计更新必须幂等，带 floor 保护（`Math.max(0, …)`）——不允许出现负数。
- 队列/后台任务必须幂等：重复处理不得重复计数或生成重复向量。
- 一致性敏感流程优先使用数据库事务或实体级互斥锁。
- 后端跨模块复用默认通过拥有方模块的 `public/*` 出口，禁止新增跨模块 deep import。
- `public/*` 出口必须按能力拆分，避免重新长成新的 mega barrel。
- 避免单文件/函数超过 ~400 行；优先拆分为小的可组合函数。

## 架构门禁

- 后端模块边界相关改动提交前，至少执行 `pnpm architecture:check`。
- 提交 PR 时，默认说明新增跨模块复用是否已经收口到 `public/*`。

## 配置与默认值

- 基础设施（连接、密钥、provider 选择、feature flags、cron）→ `packages/server/src/core/config/env/schema.ts` + 仓库根目录 `.env.example`。
- 业务常量（批次大小、TTL、阈值、超时、重试策略）→ `packages/server/src/core/config/defaults/*.defaults.ts`，使用 `as const` 导出。
- `packages/server/src/core/config/env/configs.ts` 合并 env 与 defaults；配置对象通过 `packages/server/src/core/config/env.ts` 对外暴露。
- 新增环境变量：同步更新仓库根目录 `.env.example` + Zod schema。新增业务默认值：写入对应的 `*.defaults.ts`。

## 测试

- 涉及计数器/向量/存储的变更，至少需要一个集成测试覆盖增量更新、幂等性和失败回滚。
